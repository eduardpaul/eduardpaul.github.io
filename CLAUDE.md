# CLAUDE.md — Eduard Paul Lakida's CV Site

## Project overview

Static portfolio/CV site for Eduard Paul Lakida. Deployed to GitHub Pages at `https://eduardpaul.work`.

Built with Gatsby 5 (SSG), React 18, Tailwind CSS v4, and MDX. All professional profile data lives in `content/cv/cv.json` (Manfred format). Blog posts are MDX files in `content/blog/`. The site is fully static — no backend, no API calls at runtime.

---

## Commands

```bash
npm run develop   # dev server at http://localhost:8000 with hot reload
npm run build     # production build → public/
npm run serve     # serve public/ at http://localhost:9000
npm run clean     # delete .cache/ and public/ (required after config changes)
npm run deploy    # gatsby build + gh-pages -d public → pushes to gh-pages branch
npm run format    # prettier across all JS/JSX/JSON/MD files
```

**When to run `gatsby clean`:** always after changing `gatsby-config.mjs`, `gatsby-node.js`, or adding/removing plugins. Stale cache causes confusing build errors.

## Architecture

### Data flow

```
content/cv/cv.json
  └─ gatsby-transformer-json → CvJson node
        └─ gatsby-node.js (onCreateNode)
              downloads remote images via createRemoteFileNode
              └─ CvJsonEnhanced node  ← always use this for images
                    imageObject.localFile.childImageSharp.gatsbyImageData
                    imageObject.localFile.publicURL  (SVG fallback)

content/blog/**/*.mdx
  └─ gatsby-plugin-mdx → Mdx node
        └─ gatsby-node.js adds fields.slug
              └─ src/templates/blog-post.js renders each post
```

### Pages

| Route | File |
|-------|------|
| `/` | `src/pages/index.js` |
| `/about` | `src/pages/about.js` |
| `/activity` | `src/pages/activity.js` |
| `/deepinfant` | `src/pages/deepinfant.js` (iframe embed) |
| `/[post-slug]` | `src/templates/blog-post.js` (generated by gatsby-node) |

### Key components

- `Layout` — shell with sticky header, optional author card (non-root pages), footer
- `Hero` — full-width bio with avatar from `cvJsonEnhanced`
- `Activity` — FlexSearch-powered post grid (homepage preview, not the full page)
- `Certifications` — certification card grid from `knowledge.studies` filtered by `studyType === 'certification'`
- `Experience` — timeline from `experience.jobs[].roles[]`
- `Highlights` — artifact cards from `experience.publicArtifacts`
- `Competencies` — fact cards from `aboutMe.interestingFacts`
- `AnimatedSection` — IntersectionObserver-based fade-in wrapper
- `ReadMore` — mobile-only content truncation with expand/collapse
- `CvImage` — handles GatsbyImage / SVG publicURL / external URL fallbacks

## cv.json data model (Manfred format)

Top-level keys: `$schema`, `settings`, `aboutMe`, `experience`, `knowledge`, `careerPreferences`, `manfredSpecificData`

Key paths used by components:

```
aboutMe.profile.{name, surnames, title, description, avatar.{link, alt}}
aboutMe.interestingFacts[].{topic, fact}           → Competencies
experience.jobs[].{organization.name, roles[].{name, startDate, finishDate, challenges[].description}}  → Experience
experience.publicArtifacts[].{details.{name, description, URL, image}, relatedCompetences[], type, publishingDate}  → Highlights
knowledge.studies[].{studyType, name, startDate, institution.{name, URL, image}}  → Certifications
careerPreferences.contact.publicProfiles[].{type, URL}  → Footer
```

All image objects have the shape `{ link: string, alt: string }`. After `gatsby-node.js` processes them, they also have `localFile___NODE` → `localFile { childImageSharp { gatsbyImageData }, publicURL }`.

## CvJson vs CvJsonEnhanced

**Always use `cvJsonEnhanced` when you need images.** `cvJson` is the raw data without downloaded images — `localFile` will be null.

`CvJsonEnhanced` is created in `gatsby-node.js` `onCreateNode` and is a deep clone of `CvJson` with `localFile___NODE` fields injected on every object that has a `link` key. It is not available in the development server on the very first cold start — run `gatsby build` or wait for `gatsby develop` to finish bootstrapping.

```graphql
# CORRECT — use cvJsonEnhanced for anything with images
cvJsonEnhanced {
  aboutMe { profile { avatar { alt link localFile { childImageSharp { gatsbyImageData(width: 400, placeholder: BLURRED) } } } } }
}

# OK — use cvJson for text-only data (faster query, no image overhead)
cvJson {
  experience { jobs { organization { name } roles { name startDate finishDate } } }
}
```

## Image handling

Always render images through `CvImage`. Never use `<img src={imageObject.link}>` directly.

```jsx
import CvImage from './core/cvimage';
<CvImage imageObject={cert.institution.image} className="h-24 object-contain" />
```

`CvImage` priority:
1. `GatsbyImage` from `localFile.childImageSharp.gatsbyImageData` (raster images — optimized)
2. `<img src={localFile.publicURL}>` (SVG files — sharp can't process these)
3. `<img src={imageObject.link}>` (fallback if download failed at build time)

For the OG/Twitter social image (`siteMetadata.image`), put the file in `static/` and reference it as `/filename.svg`. Files under `src/images/` are processed by webpack and get content-hashed filenames — they are not accessible at a stable URL.

## JSX rules — critical

### Always use `className`, never `class`

```jsx
// WRONG — silently drops all Tailwind styles in JSX
<div class="flex items-center gap-4">

// CORRECT
<div className="flex items-center gap-4">
```

### SVG attributes must be camelCase in JSX

```jsx
// WRONG
<svg stroke-width="2" stroke-linecap="round">

// CORRECT
<svg strokeWidth="2" strokeLinecap="round" strokeLinejoin="round">
```

### Keys go on the outermost element of a `.map()` return

```jsx
// WRONG — React can't reconcile
posts.map(post => (
  <AnimatedSection>
    <Link key={post.slug} ...>

// CORRECT
posts.map(post => (
  <AnimatedSection key={post.slug}>
    <Link ...>
))
```

## SSR rules — Gatsby pre-renders every page

Gatsby runs component code in Node.js during build. `window`, `document`, `navigator`, and `localStorage` are undefined at build time.

### Never read `window` at module level or in `useState` initializers

```js
// WRONG — causes build crash or hydration mismatch
const [width, setWidth] = useState(window.innerWidth);
const [width, setWidth] = useState(typeof window !== 'undefined' ? window.innerWidth : 0);
//   ^ The second form looks safe but causes a hydration mismatch:
//     server renders with 0, client's useState initializer runs BEFORE hydration
//     and sees window.innerWidth = 375 on mobile → structural mismatch.

// CORRECT — always start with 0/null, update in useEffect after mount
const [width, setWidth] = useState(0);
useEffect(() => {
  const handler = () => setWidth(window.innerWidth);
  window.addEventListener('resize', handler);
  handler();
  return () => window.removeEventListener('resize', handler);
}, []);
```

### Keep IntersectionObserver options stable

Passing an inline object to `useIntersectionObserver` creates a new reference every render, causing the observer to be torn down and recreated — re-triggering animations on every state update.

```js
// WRONG — new object reference each render
const AnimatedSection = ({ children }) => {
  const [ref, isIntersecting] = useIntersectionObserver({ threshold: 0.1 });

// CORRECT — module-level constant, stable reference
const OBSERVER_OPTIONS = { threshold: 0.1 };
const AnimatedSection = ({ children, className = '' }) => {
  const [ref, isIntersecting] = useIntersectionObserver(OBSERVER_OPTIONS);
```

## Tailwind CSS v4

This project uses **Tailwind v4** (`tailwindcss: 4.3.0`) with `@tailwindcss/postcss`. The v4 API is different from v3:

- Config is minimal (`tailwind.config.js` only sets `content` paths)
- The `@tailwind base/components/utilities` directives are replaced with `@import "tailwindcss"` in CSS
- `@tailwindcss/typography` provides the `prose` class for blog post content
- Do not install or reference v3-specific plugins

Avoid putting conflicting utility classes on the same element:

```jsx
// WRONG — max-w-none and max-w-3xl both set max-width; one is dead
<article className="prose max-w-none mx-auto max-w-3xl">

// CORRECT
<article className="prose mx-auto max-w-3xl">
```

## FlexSearch / search results normalization

`useFlexSearch` returns flat objects (fields at the top level). GraphQL queries return nested `frontmatter` / `fields` objects. Always normalize:

```js
const title = post.frontmatter?.title || post.title
const slug = post.fields?.slug || post.slug
const external = post.frontmatter?.external || post.external
const description = post.frontmatter?.description || post.description || post.excerpt
const date = post.frontmatter?.date || post.date
```

### External post routing

Posts with an `external` frontmatter field should open the external URL directly, not navigate to an internal Gatsby page:

```jsx
external ? (
  <a href={external} target="_blank" rel="noopener noreferrer" className={cardClass}>
    {cardContent}
  </a>
) : (
  <Link to={slug} className={cardClass}>
    {cardContent}
  </Link>
)
```

## Blog post MDX frontmatter

Required fields:

```yaml
---
title: Post title
date: "2024-02-29T00:00:00.000Z"   # ISO 8601
description: "One-sentence summary shown in post cards."
---
```

Optional fields:

```yaml
external: https://medium.com/...    # if post is cross-published; cards link to external URL
```

Images inside posts go in the same folder as the `.mdx` file and are referenced with a relative path (`./image.png`). They are processed by `gatsby-remark-images`.

## Gatsby config notes

### Plugin ordering matters

- `gatsby-plugin-offline` must be **last** in the `plugins` array (it wraps the service worker around everything else)
- `gatsby-remark-prismjs` must be **last** in `gatsbyRemarkPlugins` inside `gatsby-plugin-mdx`

### ESM vs CJS

- `gatsby-config.mjs` uses ES Modules (`export default`, `import`)
- `gatsby-node.js` uses CommonJS (`require`, `module.exports`) — Gatsby 5 supports both but they must not be mixed within a file

### RSS feed

The feed uses `node.body` (the raw MDX string) for `content:encoded`. `node.html` does not exist on MDX nodes. The `body` field must be explicitly requested in the feed GraphQL query.

## Component data fetching conventions

Components fetch their own data via `useStaticQuery`. Page-level `pageQuery` exports should only query data the page component itself uses directly — not data consumed internally by child components.

```js
// WRONG — Activity has its own useStaticQuery; this is dead weight in index.js
export const pageQuery = graphql`{ localSearchPages { index store } allMdx { nodes { ... } } }`

// CORRECT — only what the page component actually reads
export const pageQuery = graphql`{ site { siteMetadata { title } } }`
```

## SEO component

`src/components/seo.js` is used as the Gatsby Head API export on every page:

```jsx
export const Head = ({ location }) => (
  <Seo title="Page Title" pathname={location.pathname} />
)
```

Props: `title`, `description`, `pathname`, `image`, `keywords` — all optional.

The component emits: `<title>`, description, keywords, OG tags, Twitter card tags, and a canonical link.

**Keywords** are built automatically from `cvJson` (name + job title + main stack + all hard skills, deduplicated). Individual pages can pass a custom `keywords` prop to override the default for that page — useful for blog posts:

```jsx
export const Head = ({ data: { mdx: post } }) => (
  <Seo
    title={post.frontmatter.title}
    description={post.frontmatter.description}
    keywords="certificate pinning, WCF, .NET 8, C#, SOAP security"
  />
)

## Static assets

Files in `static/` are copied verbatim to `public/` at build time with no processing. Use `static/` for:
- Files that must be at a predictable URL (e.g., `/favicon.svg` for OG images)
- The DeepInfant demo app (`static/deepinfant-app/`)
- `CNAME` (GitHub Pages custom domain)

Files in `src/images/` are processed by `gatsby-plugin-sharp` and `gatsby-plugin-image` — they get content-hashed URLs and are not accessible at a stable path.

---

## 404 page

`src/pages/404.js` is intentionally minimal (no Layout wrapper, no Tailwind). Do not add the full site layout to it — Gatsby's offline plugin uses this page as a fallback shell.

## Things to avoid

- **Do not** use `class=` instead of `className=` in JSX — styling silently breaks
- **Do not** use HTML attribute names in SVG JSX (`stroke-width`, `fill-rule`) — use camelCase
- **Do not** read `window`/`document` outside `useEffect` — breaks SSR build
- **Do not** pass inline object literals as `useEffect` dependencies — causes infinite loops
- **Do not** put conflicting Tailwind utilities on the same element (`max-w-none max-w-3xl`)
- **Do not** use `cvJson` when you need image data — use `cvJsonEnhanced`
- **Do not** reference `node.html` in the RSS feed config — MDX nodes have `body`, not `html`
- **Do not** hardcode the `keywords` meta tag content — it is derived from `cvJson` automatically; update `cv.json` instead
- **Do not** change the deploy target from `public` back to `build`
- **Do not** import from `src/components/core/socialicon.js` — file was deleted (was never used; footer uses lucide-react icons directly)
